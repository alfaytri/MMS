Context

We have an admin-driven user management system (create/edit/reset password) using Supabase + Next.js middleware. The system is functionally correct, but needs security and consistency hardening before implementation.

1. Enforce Authorization (CRITICAL)
Problem

Current API routes only check if the user is authenticated.
They do NOT verify if the user is an admin.

Risk

Any authenticated user could:

create users
reset passwords
modify roles
Required Fix

Add strict authorization to all admin routes:

Affected routes:

POST /api/users/create
PATCH /api/users/[id]
POST /api/users/reset-password
Implementation requirement
Add isAdmin(user) check using:
JWT role claim OR
database role lookup (preferred)
Expected behavior
if (!isAdmin(user)) {
  return new Response("Forbidden", { status: 403 })
}
2. Handle JWT Metadata Staleness
Problem

Middleware depends on:

session.user.user_metadata.must_change_password

But JWT is not refreshed automatically after:

user creation
admin password reset
user password change
Risk
User may bypass /change-password temporarily
Middleware enforcement becomes inconsistent
Required Fix
Option A (required minimum)

Force session refresh after updates:

await supabase.auth.refreshSession()
Option B (recommended fallback)

If metadata is missing or stale:

fallback to DB check (profiles.must_change_password) in middleware
Goal

Ensure immediate enforcement consistency.

3. Role Update Atomicity
Problem

Current flow:

DELETE roles
INSERT roles

No transaction → partial failure possible.

Risk

User may end up with:

zero roles
inconsistent permissions
Required Fix

Implement Postgres RPC:

BEGIN;
DELETE FROM user_custom_roles WHERE user_id = $1;
INSERT INTO user_custom_roles (...) VALUES (...);
COMMIT;

Call via Supabase RPC from API.

Goal

Make role updates atomic and failure-safe.

4. Secure Password Handling (Admin-set passwords)
Problem

Admins manually assign passwords.

Risks
insecure sharing (plain text)
weak passwords
no audit visibility
Required Fix
Minimum:
Strengthen validation (not just ≥8 chars + digit)
Log password reset events (audit table or logging system)
Optional (future-ready):
mark passwords as temporary
enforce immediate reset (already partially implemented)
5. Add Rate Limiting
Problem

Sensitive endpoints are unprotected:

/api/users/create
/api/users/reset-password
Risk
abuse
brute-force or spam user creation
Required Fix

Implement rate limiting:

per user OR per IP
middleware or API-layer enforcement
6. Prevent Self-Deactivation
Problem

Admin can deactivate their own account.

Risk
loss of admin access
system lockout
Required Fix
if (targetUserId === currentUserId && is_active === false) {
  return error("You cannot deactivate yourself")
}
7. Harden Middleware Path Allowlist
Problem

Current bypass logic is implicit.

Risk

Unintended routes may bypass enforcement.

Required Fix

Define explicit allowlist:

const allowedPaths = [
  '/login',
  '/change-password',
  '/api/users/me/change-password',
]

const isAllowed =
  allowedPaths.includes(path) ||
  path.startsWith('/api/auth/')
Goal

Prevent accidental bypass of password enforcement.

8. Clarify Source of Truth (Design Note)
Current
JWT → runtime enforcement
DB → UI visibility
Risk

Potential drift between JWT and DB.

Required Action
Document clearly that:
JWT is enforcement layer
DB is mirrored state
Ensure all API routes dual-write consistently

(No immediate code change required, but must be enforced discipline.)
