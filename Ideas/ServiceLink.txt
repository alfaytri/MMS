🔗 Service Links — Page Working & Cross-Module Connections
Route: /master-data/services → Inventory tab → Service Links sub-tab File: src/domains/services/components/ServiceInventoryLinksView.tsx (+ serviceInventoryHelpers.ts, useServiceInventory.ts) Table: service_inventory (junction between services ↔ inventory_items)

1️⃣ What This Page Does (One-Liner)
It's the matrix that decides what inventory items each service consumes, sells, or installs — the bridge that makes services aware of stock so the system can auto-deduct, auto-charge, and auto-warranty parts when an order/visit is completed.

2️⃣ Top Bar — Inch by Inch
🔹 Header Counters (left)
15 services    ✅ 3 linked    ⚠️ 12 unlinked
15 services — total leaf services (only leaves, not categories) collected via collectLeaves() from all 3 service trees (normal, contract, mobile).
✅ 3 linked — leaf services that have ≥1 row in service_inventory.
⚠️ 12 unlinked — leaves with zero links (operational risk: no auto-deduct will happen on completion).
🔹 Toolbar (left → right)
Control	What it filters
🔍 Search services…	Substring match on nameEn / nameAr / breadcrumb path
All Types ▾	all / normal / contract / mobile — which service tree the leaf belongs to
All Statuses ▾	all / linked / unlinked
🔵 + New Link (right)	Opens the link-creation dialog (pre-selects nothing)
3️⃣ Main Table — Column Map
Column	Content	Source
Service	Service name + breadcrumb path (AC › Split › Level 1 Maintenance › Capacitor changing)	services table + parent walk
Status	⚠️ 0 (unlinked) or ✅ N (N items linked)	count(service_inventory) per service
Items	Comma-separated brand variants linked (e.g., 1 ton · Piston, Alfacool +1)	inventory_brand_variants.label
Types	Single-letter badge of link types present: C (Consumable), S (Select One), I (Install All)	LINK_TYPE_CONFIG in helpers
Qty	Sum of quantity across all linked items	service_inventory.quantity
➕ / 🗑️	Per-row add link / delete link	mutations
🔹 Expanded Row (after clicking ▾)
Each linked item shows its own editable row:

🔸 [Brand variant] [Item name · SKU] [Selling price badge]
                 [Link Type ▾] [Warranty months ▾] [Qty input] [Unit] [🗑️]
Brand variant icon (layers icon) → inventory_brand_variants
Selling price badge (e.g., QAR 500) → first active variant's selling_price (used for select_one / install_all pricing on quotations)
Link Type dropdown → updates service_inventory.link_type
Warranty months dropdown (the 12 you see) → product warranty months attached to this link, inheritable down from the item/category warranty
Qty input → service_inventory.quantity
Unit → read-only inventory_items.unit (Piece, Kg, L…)
4️⃣ The 3 Link Types (the heart of the page)
Type	Letter	Behavior on Order/Visit completion
Consumable	C	Auto-deducted from stock. NOT charged to customer. Internal usage only (rags, screws, refrigerant top-up).
Select One	S	Technician/CSR must pick ONE option from the linked items (e.g., choose 1 capacitor brand). Charged to customer at variant selling_price.
Install All	I	All linked items in the same group_label are installed together and charged. Used for kits/bundles.
→ Defined in serviceInventoryHelpers.ts → LINK_TYPE_CONFIG.

5️⃣ How This Page Connects to the Rest of the System
🔄 A. Order Creation (CreateVisit.tsx / CreateOrderQuotation.tsx)
When a CSR adds a service to an order:

The order form reads service_inventory for that service.
Consumables → silently attached, will deduct on completion.
Select One → renders a brand picker with selling_price shown.
Install All → renders grouped checkboxes by group_label; all charged. → Output stored in selected_services.selectedProducts[] (see shared/types/services.ts → SelectedProductItem).
🔄 B. Quotation Pricing
select_one / install_all items contribute to productTotal on the quotation line.
The price snapshot is taken from the brand variant at quote time (locked).
🔄 C. Order Completion → Stock Deduction
When the order moves to completed:

Edge function deduct-order-stock reads the linked items + chosen variants.
For each item it calls deduct_fifo_layers() RPC → consumes oldest FIFO layers.
Writes to inventory_stock_movements (type: order_consumption).
Writes to cogs_entries (financial ledger) using unit_cost + landed_cost_per_unit.
Updates inventory_brand_variants.stock_level via trigger.
🔄 D. Sale Orders (deduct-sale-stock)
Sale orders bypass service_inventory entirely (they're direct item sales), but use the same brand variants referenced here.
🔄 E. Warranty Generation
The warranty_months set per link feeds the product warranty on the completed visit's invoice line.
Combined with services.workmanshipWarrantyMonths to build the warranty card shown to the customer.
Multi-part warranties cascade via inventory_warranty_items.
🔄 F. Purchase Orders (Indirect)
POs replenish the brand variants linked here.
When stock for a select_one item drops to 0, the order form will warn the CSR (the system-wide "warn-but-allow" stock policy).
🔄 G. Service Reminders & Contracts
Contract visits generated from contracts reuse the same service_inventory map → consumables auto-deduct on each scheduled visit.
🔄 H. Mobile App (Customer Portal)
For services in the mobile tree, the booking engine reads linked select_one items to show the customer a parts picker (e.g., choose AC filter brand) before checkout.
6️⃣ Database Side-Effects of Editing on This Page
Action on this page	Tables touched	Downstream effect
Add link (➕)	INSERT service_inventory	New service↔item rule active immediately for new orders
Change Type	UPDATE service_inventory.link_type	Changes whether item is auto-deducted vs. charged
Change Qty	UPDATE service_inventory.quantity	Changes deduction amount per service execution
Change Warranty months	UPDATE service_inventory.warranty_months	New invoices show new warranty; old invoices unchanged
Delete (🗑️)	DELETE service_inventory	Future orders won't auto-deduct; historical orders unaffected
Trigger fires	inventory_items.linked_services_count recomputed	Drives the "Used in N services" badge on Items list
7️⃣ Why "Unlinked" Matters (the ⚠️ 12)
An unlinked service = silent inventory leak:

Technician uses parts in the field.
No deduction happens because nothing is mapped.
stock_level drifts from physical reality → discovered only via Inventory Check.
COGS is understated → margin reports lie.
That's why the status filter + counters are surfaced so prominently at the top — it's a daily ops health check.
