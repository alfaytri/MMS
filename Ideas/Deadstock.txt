💀 Dead Stock Report — Inch-by-Inch Walkthrough
Route: /purchase/dead-stock File: src/domains/purchase/pages/DeadStockReport.tsx Purpose: Surface inventory that isn't moving — so procurement/ops can liquidate, discount, or write it off before it becomes a permanent loss.

1️⃣ Page Header (top strip)
Dead & Slow-Moving Inventory
Items with no stock movements — identify aging inventory
H1 (text-lg font-semibold) — page title.
Sub-line (text-xs text-muted-foreground) — one-line definition so a new user instantly knows what they're looking at.
No global toolbar, no "+ New" button — this page is read-only analytics, not data entry.

2️⃣ Three Summary Cards (grid-cols-3)
Each card = a money bucket totalling stock value at risk for that bucket. Value is computed as stock_level × average_cost in QAR.

🟥 Card 1 — Dead Stock (180+ days)
Border: border-destructive/20 (red tint)
Icon: 💀 Skull (red)
Big number: QR <total value of all "dead" items> in red, bold
Sub: <count> items
Meaning: zero movement for 6+ months — this is money frozen on the shelf.
🟧 Card 2 — At Risk (91–180 days)
Border: border-orange-500/20
Icon: ⚠️ AlertTriangle (orange)
Big number: orange total
Meaning: approaching dead-stock threshold — last chance to discount or bundle.
🟨 Card 3 — Slow Moving (31–90 days)
Border: border-warning/20 (yellow)
Icon: 📉 TrendingDown
Big number: yellow total
Meaning: velocity dropped — watch list, may recover.
⚠️ Note: "Active" (≤30 days) items are deliberately not summarized — they're healthy. They only appear in the table if you filter for them.

3️⃣ Filter / Toolbar Strip
A single horizontal flex row:

Element	Purpose
🔍 Search box (max-w-xs)	Live-filter by item name, brand, or SKU (case-insensitive includes)
Status dropdown (w-150px)	All Statuses / Active / Slow / At Risk / Dead
Right-aligned counter	<n> items — reflects current filtered count
4️⃣ The Main Table (8 columns)
#	Column	Content	Notes
1	Item	Item name (bold) + Category (small grey under it)	From inventory_items.name_en + joined inventory_categories.name_en
2	Brand	Brand name	From joined brands.name (falls back to "Unknown")
3	SKU	The brand variant code in primary color	inventory_brand_variants.code — "—" if null
4	Qty (right-aligned)	On-hand units	stock_level
5	Value (QR) ↕	qty × average_cost	Sortable — click header to toggle asc/desc
6	Last Movement	Date of most recent stock movement, OR "Received <date>" if no movements	See logic below
7	Days Idle ↕	Integer days since reference date	Sortable — default sort, descending
8	Status	Color-coded badge: Active / Slow / At Risk / Dead	
The two ↕ sortable columns show ArrowUpDown icons; clicking the same header twice flips direction.

Empty state: a single centered row → "No items found".

5️⃣ The Workflow / Data Pipeline (what happens on mount)
When the page loads, useEffect runs four sequential queries, then merges them in memory:

Step 1 — Pull every variant with stock

SELECT id, brand_id, code, stock_level, average_cost, item_id,
       inventory_items(name_en, inventory_categories(name_en)),
       brands(name)
FROM inventory_brand_variants
WHERE stock_level > 0
Items with 0 stock can't be "dead" — they're already gone.

Step 2 — Get the last movement date per variant

SELECT brand_variant_id, created_at
FROM inventory_stock_movements
WHERE brand_variant_id IN (...)
ORDER BY created_at DESC
A Map keeps only the first (= most recent) row per variant.

Movements include: PO receival, sale deduction, transfer, adjustment, return, write-off — anything that touches stock.

Step 3 — Fallback: oldest FIFO layer date per variant

SELECT brand_variant_id, date
FROM fifo_cost_layers
WHERE brand_variant_id IN (...) AND remaining_qty > 0
ORDER BY date ASC
Used when an item was received but never moved since — there's no movement row, so we count age from when it arrived.

Step 4 — Compute daysIdle per row

referenceDate = lastMovement ?? oldestLayer
daysIdle = floor((today - referenceDate) / 1 day)
// If neither exists → 999 (treated as ancient)
Step 5 — Bucket into status

≤30  → active
≤90  → slow
≤180 → at_risk
>180 → dead
Step 6 — Render
Summary cards: reduce value totals per bucket.
Table: useMemo applies search + status filter + sort.
6️⃣ How This Page Connects to the Rest of the System
┌──────────────────────┐
│ inventory_brand_     │   ← stock & avg cost (mirror of FIFO)
│ variants             │
└──────┬───────────────┘
       │
       ▼
┌──────────────────────┐     populated by:
│ inventory_stock_     │     • process-receival       (PO inflow)
│ movements            │     • deduct-order-stock     (visit consumption)
│ (the activity feed)  │     • deduct-sale-stock      (POS sale)
└──────────────────────┘     • process-warehouse-transfer
       │                     • process-stock-adjustment
       │                     • process-return
       ▼
┌──────────────────────┐
│ fifo_cost_layers     │   ← cost layers per receival
│ (oldest layer = age  │
│  of physical stock)  │
└──────────────────────┘
Upstream feeders (anything that creates a movement row resets daysIdle):

✅ PO Receival → process-receival
✅ Sale Order / Field Visit → deduct-*-stock
✅ Warehouse Transfer → process-warehouse-transfer
✅ Stock Adjustment → process-stock-adjustment
✅ Return → process-return
Downstream actions (what users typically do from this report):

Discount campaign → Promotions module → create voucher targeting these SKUs.
Write-off → Master Data → Inventory → Brand row → Adjust → decrease / write_off (logged in stock_adjustments + inventory_stock_movements).
Transfer to a busier branch → Warehouse Transfers.
Stop reordering → Procurement filters supplier history before next PO.
7️⃣ Edge Cases & Gotchas
Situation	Behaviour
Variant has stock but zero movements AND no FIFO layer	daysIdle = 999, shows as Dead, "Last Movement = Unknown"
Variant received yesterday, never moved	Active (1 day), "Last Movement = Received <date>"
Variant fully sold then restocked	Resets — only counts since the new receival's movement row
average_cost = 0	Value column shows 0 — won't inflate summary totals (correct)
Soft-deleted items	Still pulled (no deleted_at filter) — potential noise, would be a refinement
8️⃣ Who Uses It & When
Procurement Manager — weekly review before placing new POs.
Warehouse Manager — monthly liquidation candidates.
Finance — quarterly inventory provisioning (write-down dead stock value).
Ops Director — dashboard health-check on capital tied up in non-moving inventory.
